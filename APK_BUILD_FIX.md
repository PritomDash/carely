# Android APK Build/Rebuild Instructions

## Background
Carely is a PWA served from the Vercel-hosted frontend. The Android app is not a native build in this repo — it's a **TWA (Trusted Web Activity)** wrapper generated by [PWA Builder](https://www.pwabuilder.com/), which loads the deployed web app inside a WebView-based shell.

The `Carely.apk` file attached to the GitHub release is expected to be **at least 3-5MB**. A ~900KB file is a red flag — it's most likely an unsigned/incomplete package, not a real installable TWA APK.

## Rebuild steps

1. **Confirm the PWA manifest is valid** at the deployed URL:
   - `https://<your-vercel-domain>/manifest.json` must be reachable and include `name`, `short_name`, `icons` (192x192 and 512x512 at minimum), `start_url`, `display: "standalone"`.
   - Confirm a service worker is registered (PWA Builder checks for this).

2. **Generate the package**
   - Go to https://www.pwabuilder.com/
   - Enter the production URL (e.g. `https://carely.vercel.app`)
   - Let it run the manifest/service worker/security checks — all should pass (green) before packaging.
   - Choose **Android** → **Google Play** package type (this produces a proper signed TWA, not the "test package" option).

3. **Signing**
   - PWA Builder can generate a new keystore for you, or accept an existing one.
   - **Save the generated keystore file (`.keystore`/`.jks`) and its passwords somewhere safe** — you need the same keystore for every future update, or Android will refuse to install the new APK over the old one ("app not installed as package conflicts with an existing package").
   - Do not use the "unsigned"/debug package for the public release download — it won't install on most devices without enabling developer/debug installs.

4. **Verify the downloaded APK before uploading**
   - Check file size: should be 3-5MB+, not ~900KB.
   - Optional sanity check with `aapt` (Android SDK build-tools) or `unzip -l Carely.apk` — a valid APK should contain `AndroidManifest.xml`, `classes.dex`, `resources.arsc`, and a `META-INF/` folder with signature files (`.SF`, `.RSA`/`.DSA`).

## Upload the new APK to GitHub Releases

1. Go to `github.com/PritomDash/carely/releases`.
2. Click **Draft a new release** or edit the existing `v1.0.0` release.
3. Delete the old `Carely.apk` asset.
4. Upload the new, validated APK.
5. **The filename must be exactly `Carely.apk`** — the frontend download links point at:
   ```
   https://github.com/PritomDash/carely/releases/latest/download/Carely.apk
   ```
   which resolves by filename against the *latest* release, so a mismatched filename silently 404s.
6. Publish the release.

## Frontend download links
`LandingPage.js` has three `<a href=".../Carely.apk">` links (hero section, download section, install instructions box). All three now carry a `download="Carely.apk"` attribute so browsers are hinted to save the file directly instead of navigating to it — though the actual "download vs open" behavior is ultimately governed by the `Content-Disposition` header GitHub serves on release assets.
